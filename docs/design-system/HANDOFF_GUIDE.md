# Claude Design → Claude Code handoff for Voice Diary

How to take a Claude Design output bundle and integrate it into the iOS app or the server's HTMX templates **without breaking the Style Dictionary pipeline**.

The pipeline is the load-bearing piece: token JSON in `docs/design-system/tokens/*` is the source of truth; everything else (`DSColor.swift`, `DSMetrics.swift`, `tokens.css`) is generated by `bash scripts/build_design_system.sh`. Hand-edited token outputs **will be silently overwritten** on the next build. This guide exists so that doesn't happen.

---

## The mental model

```
docs/design-system/tokens/*.json     ← single source of truth
docs/design-system/specs/components/ ← component contracts
            │
            ▼  bash scripts/build_design_system.sh
            │
            ├─► ios/Sources/DesignSystem/DSColor.swift     (generated)
            ├─► ios/Sources/DesignSystem/DSMetrics.swift   (generated)
            ├─► ios/Sources/DesignSystem/DSSemantic.swift  (generated)
            ├─► server/webapp/static/tokens.css            (generated)
            └─► server/webapp/static/tokens-semantic.css   (generated)

Hand-written aliases:
  ios/Sources/DesignSystem/Theme.swift            (Theme.color.text.primary, …)
  ios/Sources/DesignSystem/DSButtonStyle.swift    (.dsPrimary(), .dsSecondary(), …)
  server/webapp/static/style.css :root            (CSS variable aliases)

DESIGN.md (repo root) — read-only spec consumed by Claude Design.
```

**Rule of thumb:** if you find yourself typing a hex value into a Swift file or a CSS file, stop. The hex belongs in the token JSON.

## Workflow A — Claude Design produces a screen-level mock with new visual primitives

This is the common case: Claude Design has output a screen mock that uses the existing tokens and adds nothing new.

1. **Read the token map** in the output bundle. Verify every name appears in `DESIGN.md` or `docs/design-system/tokens/*`.
2. **If everything maps cleanly**, this is purely an iOS layout change. Open the relevant file in `ios/Sources/UI/...`, edit the SwiftUI views to match the mock. Use `Theme.color.*`, `Theme.spacing.*`, `Theme.radius.*`, `Theme.font.*` — no raw values.
3. **Run the app on iPhone 17 Pro simulator** (`xcodebuild -scheme VoiceDiary -destination 'platform=iOS Simulator,name=iPhone 17 Pro'`).
4. **Diff the mock** mentally against the rendered view. Fix discrepancies in spacing/radius using existing tokens before considering whether to add new ones.

If a discrepancy can't be resolved with existing tokens, see Workflow B.

---

## Workflow B — Claude Design output needs a new token

For example: Claude Design uses a "warm-50" background that doesn't exist in the current palette, or a 14px radius value between `md` and `lg`.

**Don't shoehorn a new value into Swift.** Add it through the pipeline:

1. **Decide which token group it belongs to.** Color → `tokens/color/`. Spacing → `tokens/layout/spacing.json`. Radius → `tokens/layout/radius.json`. Font → `tokens/typography/font.json`.
2. **Edit the JSON source.** Use the existing schema:
   ```json
   "warm": {
     "50": { "value": "#FAF7F2", "type": "color" }
   }
   ```
3. **If it's a semantic token** (e.g. "container-warm"), also wire it in `tokens/color/semantic.light.json` and `tokens/color/semantic.dark.json` so light/dark stay paired.
4. **Run** `bash scripts/build_design_system.sh` from the repo root. This regenerates Swift + CSS outputs and copies them into both consumer trees.
5. **If you added a semantic token**, surface it in `Theme.swift`:
   ```swift
   public struct BgColors: Sendable {
       …
       public let containerWarm = DSSemantic.Bg.container_warm
   }
   ```
   (The generator names use snake_case; the Theme alias uses camelCase.)
6. **Update `DESIGN.md`** so the next Claude Design run sees the new token. The DESIGN.md frontmatter should mirror the JSON source — both human-readable for Claude Design and a documentation artefact.
7. **Commit all of:** `tokens/*.json`, the regenerated `DSColor.swift` / `DSMetrics.swift` / `DSSemantic.swift` / `tokens.css` / `tokens-semantic.css`, the updated `Theme.swift`, and `DESIGN.md`. They form a single coherent change.

> *Why commit the generated files?* Because both apps consume them at build time, and we don't want Xcode or a CI runner to need a Node toolchain to build. The generated files ARE part of the source of truth from the consumers' perspective; they're just not edited by hand.

## Workflow C — Claude Design output needs a new component variant

For example: a "tertiary" button variant that doesn't exist in `specs/components/button.json`.

1. **Edit `docs/design-system/specs/components/button.json`** to add the variant under `variants:`. Reference existing semantic tokens; don't introduce hex values here.
2. **Update `ios/Sources/DesignSystem/DSButtonStyle.swift`** to add the matching SwiftUI ButtonStyle. The naming convention is `.dsTertiary(size:fullWidth:)`.
3. **Update `DESIGN.md` Components section** with the new variant + use case.
4. **No regeneration step needed** for component specs — they're consumed by humans (us) and by Claude Design, not by Style Dictionary.

If you find yourself adding a fourth, fifth, sixth button variant — stop. The spec was deliberately constrained to 5 variants × 3 sizes. Push back on the design instead.

---

## Workflow D — Claude Design output is a server-side template change

Voice Diary's HTMX review/admin UI lives at `server/webapp/templates/`. The CSS strategy:

- Generated tokens live at `server/webapp/static/tokens.css` (`--color-*`, `--spacing-*`).
- Semantic tokens at `server/webapp/static/tokens-semantic.css` (`--text-primary`, `--bg-container`, light + dark variants via `prefers-color-scheme`).
- A hand-written `style.css` aliases legacy names into the DS variables (e.g. `--page-bg: var(--bg-surface);`).

To integrate a server-side mock:

1. **Identify which template** Claude Design is redesigning. Most likely under `server/webapp/templates/review/` or `server/webapp/templates/admin/`.
2. **Add new CSS classes** in `style.css` that *consume* `var(--*)` from the alias layer. Never write a hex value, spacing literal, or font-size literal in HTML or in component-level CSS.
3. **Test in both color schemes** by toggling system appearance — light + dark are first-class on the server too.
4. **Don't introduce a CSS framework.** No Tailwind, no Bootstrap, no PostCSS pipelines. Plain CSS + the design tokens are the contract.

## Anti-patterns — what NOT to do

### ✗ Hardcode a hex from a Claude Design mock into Swift

```swift
// WRONG — Theme is the only path for colours.
.foregroundStyle(Color(hex: "#1570EF"))
```

```swift
// RIGHT
.foregroundStyle(Theme.color.text.link)
```

### ✗ Introduce a "design-only" stylesheet for one screen

```html
<!-- WRONG — single-purpose CSS file -->
<link rel="stylesheet" href="/static/admin-redesign.css">
```

Add the new classes to `style.css` (or split into `style.css` + `style.review.css` if it's truly reusable). Keep one alias layer; don't fork.

### ✗ Edit a generated token output by hand

```swift
// WRONG — DSSemantic.swift is generated. Edits will be lost.
public enum Bg {
    public static let container = Color(red: 0.99, green: 0.97, blue: 0.95)
}
```

Edit `tokens/color/semantic.light.json` and rerun the build script.

### ✗ Add a new color or spacing token to satisfy a single screen

If only one screen needs a value, the screen is probably wrong, not the design system. Push back.

### ✗ Ship a Claude Design mock without dark mode

The user does most reflection in the evening. Dark mode is the default during the walkthrough session. If Claude Design only produced a light mock, re-prompt for dark.

## Pre-flight checklist before merging a design change

- [ ] All colours used go through `Theme.color.*` (Swift) or `var(--*)` (CSS).
- [ ] All spacing values go through `Theme.spacing.*` or `var(--spacing-*)`.
- [ ] All radii go through `Theme.radius.*` or `var(--radius-*)`.
- [ ] All fonts go through `Theme.font.*` or the CSS family variables.
- [ ] Buttons use `DSButtonStyle` variants — no ad-hoc styling.
- [ ] Light + dark both rendered and acceptable.
- [ ] `DESIGN.md` updated if any token / component variant was added.
- [ ] If `tokens/*.json` was edited, the build script ran and generated outputs were committed.
- [ ] iPhone 17 Pro simulator screenshot attached to the PR for any visible screen change.

## Reference: existing Theme + token surface

iOS (`ios/Sources/DesignSystem/Theme.swift`):

```swift
Theme.color.text.{primary, secondary, subdued, inverse, link}
Theme.color.bg.{surface, container, containerInset, containerInsetHover, …}
Theme.color.border.{primary, secondary, subdued, destructive, solid}
Theme.color.status.{success, warning, destructive}
Theme.spacing.{xxs, xs, sm, md, lg, xl, xxl, xxxl}
Theme.radius.{sm, md, lg, xl, xxl, full}
Theme.font.{display, title, headline, body, callout, subheadline, caption, monoBody, monoCaption}
Theme.motion.{snappy, soft, bouncy}   // SwiftUI Animation values
```

Buttons:

```swift
.buttonStyle(.dsPrimary(size: .lg, fullWidth: true))
.buttonStyle(.dsSecondary())
.buttonStyle(.dsOutline())
.buttonStyle(.dsGhost())
.buttonStyle(.dsDestructive())
```

Server (`server/webapp/static/tokens.css` + `tokens-semantic.css`):

```css
var(--color-gray-{50..900})
var(--color-blue-{50..900})
var(--color-green-{50..900}) /* + red, yellow, etc. */
var(--text-{primary,secondary,subdued,inverse,link})
var(--bg-{surface,container,container-inset, …})
var(--border-{primary,secondary,subdued,destructive})
var(--status-{success,warning,destructive})
var(--spacing-{0,1,2,3,4,5,6,…})
var(--radius-{sm,md,lg,xl,2xl})
var(--font-family-{sans,mono})
```

When in doubt, grep the tokens.css file in the repo before reaching for a literal — what you want is almost certainly already there.
